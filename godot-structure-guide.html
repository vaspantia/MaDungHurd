ü§î GDScript vs C++: Which Should You Use?

üìò GDScript (Recommended for Beginners)

What is it? Python-like language built into Godot

‚úÖ PROS:
  • Super easy to learn
  • No setup needed - works instantly
  • Fast to write and test code
  • All Godot tutorials use it
  • Hot-reload (see changes instantly)
  • Perfect for learning game concepts
‚ùå CONS:
  • Slower than C++ (but fast enough for most games)
  • Only works in Godot
üëâ START HERE if you're new to game development

‚ö° C++ (Advanced Performance)

What is it? Use C++ code in Godot via GDExtension

‚úÖ PROS:
  • Maximum performance
  • Reuse existing C++ libraries
  • Industry-standard language
  • Great for complex AI, physics
‚ùå CONS:
  • Complex setup required
  • Harder to learn
  • Slower development time
  • Need to compile code (slow)
  • More room for crashes/bugs
üëâ USE LATER when you need extra performance

‚ö° Setting Up C++ in Godot (GDExtension)

‚ö†Ô∏è Before You Start: Make sure you're comfortable with GDScript first! C++ is much harder.
1 Install Requirements
  • Download Visual Studio 2022 (Community - free)
  • Install "Desktop development with C++"
  • Install CMake (cmake.org)
  • Install Python 3.x (already have it)
2 Get GDExtension Template

‚Üë Downloads C++ bindings for Godot
3 Project Structure with C++

üìÅ src/ Your C++ code
üìÑ player.h Header file
üìÑ player.cpp Implementation
üìÑ SConstruct Build script
üìÅ bin/ Compiled .dll files
4 Create Player in C++
üìÑ player.h (Header File)

#define PLAYER_H
#include <godot_cpp/classes/character_body2d.hpp>
‚Üë Import Godot's CharacterBody2D class

‚Üë Inherit from CharacterBody2D (like GDScript "extends")
GDCLASS(Player, CharacterBody2D)
‚Üë Tell Godot this is a custom class

private:

public:

};
#endif
üìÑ player.cpp (Implementation)
#include "player.h"
#include <godot_cpp/core/class_db.hpp>

void Player::_physics_process(double delta) {
Vector2 velocity = get_velocity();
‚Üë Get current velocity
float direction = Input::get_axis("ui_left", "ui_right");
velocity.x = direction * speed;
set_velocity(velocity);
move_and_slide();
}
5 Compile the C++ Code
scons platform=windows
‚Üë Creates .dll file in bin/ folder
6 Use in Godot
  • Create .gdextension file to load your C++ class
  • In Godot, create node and select your C++ "Player" class
  • Works exactly like GDScript nodes!

‚úÖ 5 lines, instant testing

C++ Version

// player.h (20 lines)
// player.cpp (30 lines)
// Build script
// Compile time: 2-5 minutes
// .gdextension config

‚ö†Ô∏è 50+ lines, 5min compile
‚úÖ 2-3x faster performance
üéØ When to Actually Use C++:
  • Heavy calculations - AI pathfinding for 1000+ units
  • Complex physics - Custom physics simulations
  • Existing libraries - Want to use C++ networking library
  • Mobile performance - Need every bit of speed

‚ùå Don't use C++ for:
  • UI code - GDScript is perfect
  • Simple game logic - Not worth the complexity
  • Prototyping - Way too slow

üèÉ Player Scene Structure

Node Hierarchy Explained:

CharacterBody2D Root - handles physics & movement
‚îú‚îÄ CollisionShape2D Defines hitbox for collisions
‚îú‚îÄ AnimatedSprite2D Player graphics & animations
‚îú‚îÄ Camera2D Follows player
‚îú‚îÄ Components (Node) Container for reusable logic
‚îÇ ‚îú‚îÄ HealthComponent Manages HP, death
‚îÇ ‚îú‚îÄ DamageComponent Handles damage dealing (Area2D)

‚îú‚îÄ FootstepAudio Walking sounds

üí° Why this structure?
  • CharacterBody2D root = built-in physics movement
  • Components folder = reuse logic across different characters
  • Separate Audio node = organize multiple sound players

Player Script (player.gd):

üì¶ SECTION 1: Inheritance
extends CharacterBody2D
‚Üë This makes your script control a CharacterBody2D node (physics body)
üî¢ SECTION 2: Constants (Never Change)
const SPEED = 300.0
‚Üë Player moves 300 pixels per second
const JUMP_VELOCITY = -400.0
‚Üë Negative = jumps upward (Y axis flipped in Godot)
‚öôÔ∏è SECTION 3: Exported Variables (Editable in Inspector)
@export var max_health: int = 100
‚Üë You can change this in Godot editor without editing code!
@export var damage: int = 10
üîó SECTION 4: Node References (Connect to Scene Nodes)
@onready var sprite = $AnimatedSprite2D
‚Üë Finds the AnimatedSprite2D child node when scene loads
@onready var health_component = $Components/HealthComponent
‚Üë $ means "find child node" - / goes deeper in tree
üìä SECTION 5: State Variables
var is_attacking: bool = false
‚Üë Tracks if player is currently attacking
üé¨ SECTION 6: Lifecycle Functions
func _ready():
‚Üë Runs ONCE when scene starts
health_component.health_changed.connect(_on_health_changed)
‚Üë Listen for signal from health component

func _physics_process(delta):
‚Üë Runs 60 times per second - use for movement!
handle_movement(delta)
move_and_slide()
‚Üë Actually moves the character using velocity
üéÆ SECTION 7: Custom Functions
func handle_movement(delta):
var direction = Input.get_axis("ui_left", "ui_right")
‚Üë Returns -1 (left), 0 (nothing), or 1 (right)
velocity.x = direction * SPEED
‚Üë Set horizontal speed based on input

func _on_health_changed(new_health):
‚Üë Called when health component emits signal
print("Health: ", new_health)
üìö Code Order = Best Practice:
  1. extends ‚Üí What type of node this is
  2. const ‚Üí Values that never change
  3. @export ‚Üí Values you can edit in inspector
  4. @onready ‚Üí References to child nodes
  5. var ‚Üí Variables that change during gameplay
  6. _ready() ‚Üí Setup code
  7. _physics_process() ‚Üí Movement & physics
  8. Custom functions ‚Üí Your game logic

üìù Naming Conventions - Why They're Different:

üìÅ Files & Folders
snake_case
player_controller.gd
enemy_spawner.tscn
health_component.gd
‚Üë Easy to read, works on all OS, avoids spaces
üé® Classes
PascalCase
class_name PlayerController
class_name EnemySpawner
class_name HealthComponent
‚Üë Matches industry standard (C++, C#, Java)
üìä Variables & Functions
snake_case
var player_health: int
var is_jumping: bool
func take_damage(amount):
func calculate_distance():
‚Üë Python-style (GDScript is Python-like)
üîî Signals
snake_case
signal health_changed
signal player_died
signal item_collected
‚Üë Same as variables - they're like special variables
üî¢ Constants
UPPER_CASE
const MAX_SPEED = 500
const JUMP_FORCE = -400
const GRAVITY = 980
‚Üë SCREAMS "DON'T CHANGE ME!" - easy to spot
‚öôÔ∏è Enums
PascalCase + UPPER_CASE
enum State { IDLE, RUNNING, JUMPING }
enum Direction { NORTH, SOUTH, EAST, WEST }
‚Üë Enum name = PascalCase, values = UPPER_CASE

ü§î Why Are Variables & Functions Both snake_case?

Short Answer: GDScript follows Python style, where everything is lowercase with underscores.

‚ùå BAD - Can't Tell Them Apart
health_changed ‚Üê Is this a variable or signal?
take_damage ‚Üê Is this a function or variable?
‚úÖ GOOD - Context Makes It Clear
signal health_changed ‚Üê Signal keyword!
var health_changed = false ‚Üê var keyword!
func take_damage(): ‚Üê func keyword + ()
üí° Pro Tips for Clarity:
  • Booleans: Start with is_, has_, can_
    var is_alive, var has_key, can_jump
  • Private/Internal: Start with _underscore
    var _internal_timer, func _calculate_something()
  • Signal handlers: Start with _on_
    func _on_button_pressed(), func _on_health_changed()
  • Nodes: Descriptive names
    @onready var health_bar = $UI/HealthBar

üìã Real Example - All Together:

extends CharacterBody2D
class_name Player ‚Üê PascalCase

signal health_changed(new_health) ‚Üê snake_case
signal player_died ‚Üê snake_case

const MAX_HEALTH = 100 ‚Üê UPPER_CASE
const SPEED = 300.0 ‚Üê UPPER_CASE

enum State { IDLE, RUNNING, JUMPING } ‚Üê Pascal + UPPER

@export var max_speed: float = 500.0 ‚Üê snake_case
var current_health: int ‚Üê snake_case
var is_alive: bool = true ‚Üê snake_case with is_
var _internal_timer: float ‚Üê snake_case with _ (private)

@onready var sprite = $AnimatedSprite2D ‚Üê snake_case

func take_damage(amount: int): ‚Üê snake_case
current_health -= amount
health_changed.emit(current_health)
if current_health <= 0:
_die()

func _die(): ‚Üê snake_case with _ (private)
is_alive = false
player_died.emit()

func _on_health_changed(new_health): ‚Üê snake_case, _on_ prefix
print("Health is now: ", new_health)

üîß Component-Based Design

What is it? Breaking your game into small, reusable pieces instead of one giant script.

ü§î Why Use Components?

Reusable
+
Easy to Test
+
Mix & Match
=
Fast Development

Example: HealthComponent.gd

üì¶ Inheritance & Class Name
extends Node
‚Üë Component = plain Node (no physics, just logic)
class_name HealthComponent
‚Üë Now you can reuse this in any scene!
üîî Signals - Tell Others What Happens
signal health_changed(new_health: int)
‚Üë Fires when health changes - UI can listen to this
signal died()
‚Üë Fires when health = 0 - Player can listen to this
‚öôÔ∏è Configurable Variables
@export var max_health: int = 100
‚Üë Set different max health for player vs enemies
var current_health: int
‚Üë Tracks current health value
üé¨ Setup
func _ready():
current_health = max_health
‚Üë Start at full health
‚öîÔ∏è Take Damage Function
func take_damage(amount: int):
current_health = max(0, current_health - amount)
‚Üë Subtract damage, but never go below 0
health_changed.emit(current_health)
‚Üë Tell everyone health changed (UI updates!)
if current_health == 0:
died.emit()
‚Üë Tell everyone we died
‚ù§Ô∏è Heal Function
func heal(amount: int):
current_health = min(max_health, current_health + amount)
‚Üë Add health, but don't exceed max
health_changed.emit(current_health)
‚Üë Update UI

üéØ How to Use This Component

1Ô∏è‚É£ Add to Player Scene

Player
‚îî‚îÄ HealthComponent

In Inspector: Set max_health = 100

2Ô∏è‚É£ Add to Enemy Scene

Enemy
‚îî‚îÄ HealthComponent

In Inspector: Set max_health = 30

üìú Use in Parent Script
# In player.gd
@onready var health = $HealthComponent

func _ready():
health.died.connect(_on_player_died)
‚Üë Listen for death signal

func hit_by_enemy():
health.take_damage(10)
‚Üë Call component function

func _on_player_died():
print("Game Over!")
üí° Component Benefits:
  • ‚úÖ Write once, use everywhere - Player, enemies, destructible objects all use same component
  • ‚úÖ Easy to test - Test health logic separately from player movement
  • ‚úÖ Modular - Add/remove features by adding/removing components
  • ‚úÖ No code duplication - One place to fix bugs

üé® More Component Ideas:

  • DamageComponent - Handles dealing damage to others (Area2D that hurts things)
  • MovementComponent - Handles acceleration, friction, max speed
  • InventoryComponent - Manages items collection
  • StateMachineComponent - Manages states (idle, running, jumping)

üéØ Autoload (Singletons)

What is Autoload? Scripts that load when game starts and exist in every scene.

Normal Script vs Autoload

Normal Script
Exists only in its scene
Dies when scene changes
Autoload (Singleton)
Exists everywhere
Never dies
Access from any script

üåê Common Autoload Systems

üéÆ GameManager.gd

extends Node

‚Üë Access anywhere: GameManager.add_score(10)

‚Üë Use: SaveSystem.save_game({"score": 100})
üì° EventBus.gd
# Global event system (signals anyone can listen to)
extends Node

‚Üë Emit: EventBus.coin_collected.emit(1)

üìã How to Set Up Autoload
  1. Create script (e.g., game_manager.gd)
  2. Go to Project ‚Üí Project Settings ‚Üí Autoload
  3. Add script path: res://scripts/autoload/game_manager.gd
  4. Set name: GameManager
  5. Click Add
  6. Now use anywhere: GameManager.score = 100
‚ö†Ô∏è Warning: Don't Overuse Autoloads!
‚úÖ GOOD - Use Autoload For:
  • Game state (score, lives)
  • Audio system
  • Save/load system
  • Global events
  • Settings/config
‚ùå BAD - Don't Use For:
  • Player logic (use scene)
  • Enemy AI (use scene)
  • Level-specific code
  • UI elements
  • "Just because it's convenient"

Why? Too many autoloads = hard to debug, memory waste, tightly coupled code

üìã Script Structure Order (Always Follow This!)

1. extends
‚Üì
2. class_name
‚Üì
3. signals
‚Üì
4. constants
‚Üì
5. @export variables
‚Üì
6. regular variables
‚Üì

üé¨ Scene vs Script - When to Use What?

‚úÖ Scene + Script Together

Use when: The code controls a visual thing

  • üèÉ Characters - player.tscn + player.gd
  • üëæ Enemies - enemy.tscn + enemy.gd
  • üì¶ Items - coin.tscn + coin.gd
  • üéØ Interactive objects - door.tscn + door.gd
  • üñºÔ∏è UI with logic - menu.tscn + menu.gd
Each scene is self-contained and reusable

‚úÖ Script Only (No Scene)

Use when: Pure logic, no visuals

  • üîß Utility functions - math_helpers.gd
  • üåê Autoload singletons - game_manager.gd
  • üìä Data classes - inventory_item.gd
  • ‚ù§Ô∏è Components - health_component.gd
  • ü§ñ State machines - state_machine.gd
Logic that gets attached to other scenes
üéØ Rule of Thumb: If you can see it = Scene + Script. If it's just logic = Script only.
